RTS_PCG Game: Strategy Game Development Report Based on Procedural Content Generation

Student Name: Pu Dianheng

Student ID: 25114122G

1. Project Overview

RTS_PCG is a strategy simulation game that integrates Procedural Content Generation (PCG) technology. Players begin on a dynamically generated, resource-rich island, tasked with establishing their village from the ground up. The core mechanics center on resource gathering and villager management. Players direct their villagers to resource points by using a mouse drag-and-drop mechanism, allowing for the efficient collection of vital resources like food, wood, and stone. The long-term objective is to strategically expand the settlement and ultimately build a prosperous civilization.

2. Design and Iteration

2.1 Initial Inspiration

The creative inspiration for this project is primarily drawn from classic RTS (Real-Time Strategy) titles such as Foundation and Age of Empires. The core appeal of these games lies in resource management, base planning, and unit scheduling. Building upon this deep strategic gameplay, I utilized the new Procedural Content Generation (PCG) framework in Unreal Engine 5.6 (UE5.6) to break the limitations of static maps, creating a unique dynamic game world for every session.

2.2 Core Objectives

My development work centered around the following core objectives:

  1. Cornerstone of a Dynamic World: The PCG system is not merely a map generator but the “operating system” of the game world. The generated terrain, resource distribution, and key locations serve as underlying data that directly drive the AI behavior tree, navigation system, and game logic, realizing a truly “interactive” dynamic environment.
  2. Technology Seamlessly Serving Gameplay: I strictly adhered to the “Gameplay First” design principle. PCG technology is a powerful yet invisible tool intended to enhance rather than dominate the experience. The goal is for players to immerse themselves in the strategic fun of exploration, decision-making, and development, perceiving a natural, credible world rather than a cold technical demo. PCG acts as a silent director, meticulously setting the stage for each session, while the drama on stage—the core gameplay—remains led by the player.
  3. Achieving Infinite Replayability: By ensuring both the randomness and rationality of map generation through PCG algorithms, the game is infused with lasting vitality, encouraging players to constantly attempt new strategies and layouts to cope with unknown environmental challenges.

2.4 Asset Sources

This project’s art style and core mechanics significantly reference and benefit from the Cropout sample project released by Epic Games.


(Source: Epic Games Marketplace - Cropout Sample Project)
  • Art Style Definition: Cropout provided a complete set of low-poly assets for buildings, characters, vegetation, and resources, establishing a clear, bright, and efficient visual direction for this project.
  • Core Logic Reference: I deeply analyzed and adapted its mature villager AI, resource collection chains, and construction logic, providing a solid and verified foundation for my system architecture.
  • Deep Secondary Development & Innovative Adaptation: Building on the solid foundation of Cropout, this project underwent deep customization and functional expansion. The core contribution lies in completely refactoring the static map using UE5.6’s PCG framework, upgrading the fixed scene into infinite dynamic possibilities, successfully merging classic RTS gameplay with high replayability enabled by PCG technology.

3. PCG System Architecture & Implementation

This section details the technical implementation of the Procedural Content Generation (PCG) system, focusing on how the foundational structure of the game world is generated from scratch.

3.1 Overall PCG Workflow Architecture

The project constructs the complete game environment through the collaborative work of two core PCG graph files.


(Blueprint/PCG/Land.uasset: Responsible for generating harvestable resources (trees, shrubs, rocks) and grass vegetation on the continent.)

(Blueprint/PCG/Ocean.uasset: Responsible for generating underwater rocks and seagrass surrounding the continent.)

3.2 Detailed Analysis of Key PCG Modules

Logic Flow:

Taking shrub generation as an example, the PCG node processing flow is as follows:

Node Logic Flow and Analysis:

  1. Surface Sampler
    • Function: This is the starting point of generation. It retrieves basic data from the game landscape and outputs an initial point cloud covering the surface. These points represent all possible generation locations for shrubs.
    • Design Intent: Ensures that shrubs are generated only on the terrain surface, forming the basis for all subsequent processing.
  2. Transform Points
    • Function: Receives the point cloud from the Surface Sampler and applies basic transformation operations. It is typically used to apply a random rotation (around the Z-axis) and a slight position offset to each point.
    • Design Intent: Breaks the uniform grid pattern produced by the Surface Sampler, making the orientation and position of shrubs more random and natural, avoiding artificial alignment artifacts.
  3. Auto Adjust (Projection)
    • Function: This is a crucial node for optimization and realism. It automatically adjusts the tilt of the shrub based on the surface normal at the point’s location and may filter points based on slope attributes.
    • Design Intent: Ensures that shrubs “adhere” to slopes rather than hanging vertically or clipping into the ground, significantly enhancing the naturalness of the scene.
  4. Difference
    • Function: This node performs a Boolean operation. Its function is to subtract the area occupied by Input B (point clouds of already placed trees, rocks, or other objects) from Input A (the shrub point cloud currently being processed).
    • Design Intent: Resolves model clipping issues. Through this operation, the system ensures that shrubs do not generate directly beneath tree trunks or crowns, resulting in a distinctly layered resource distribution without interference, improving both rationality and visual quality.
  5. Spawn Actor
    • Function: This is the final generation node. It receives the point cloud that has undergone all filtering and processing and spawns actual interactive Actor instances in the scene based on the specified Blueprint class (e.g., BP_Shrub) or Static Mesh.
    • Design Intent: Transforms processed data into real objects in the game world. Unlike a simple Static Mesh Spawner, using Spawn Actor allows for the generation of Blueprint objects with custom collisions and game logic (such as harvestable health points).

Logic Summary:

Shrub generation is a process of progressively refined conditional processing. Starting from basic position determination, it undergoes key steps such as naturalized distribution, terrain adaptation, and conflict resolution, ultimately producing realistic vegetation that is both randomly distributed and physically logical. This complete chain fully demonstrates the powerful advantages of PCG technology in ensuring the rationality of resource distribution.

4. Interactive Actor Blueprint System

This section focuses on how the PCG-generated world gains “life” through a complex Blueprint system, enabling deep interaction with players and AI.

4.1 Core Game Framework

  • BP_GM.uasset (Game Mode): Defines game rules, states, and flow; serves as the central commander connecting the PCG world with game logic.
  • BP_PC.uasset / BP_Player.uasset (Player Controller / Character): Handles player input commands and translates them into interactive behaviors within the PCG-generated world.

4.2 Resource Collection System

I created fully functional interactive Actor Blueprints for every resource type generated by the PCG system.

  • Universal Interface: BPI_Resource.uasset
    • Function: Defines methods that all interactive resources must implement, such as Interact, establishing a unified interaction contract for both players and AI.
  • Resource Base Class: BP_Resource.uasset
    • Core Logic:
      • Automated Tag Management (Auto-Tagging): By reading the Resource Type (Enumerator variable), the script automatically converts it to a string and adds it to the Actor’s Tags list. This ensures all resources in the scene automatically obtain the correct type tags based on their attributes, facilitating batch retrieval or management via Get All Actors with Tag.
  • Concrete Resource Implementations:
    • BP_Tree.uasset: Responds to the “Axe” tool, drops “Wood,” and triggers villager chopping animations.
    • BP_Rock.uasset: Responds to the “Pickaxe” tool, drops “Stone,” and triggers villager mining animations.
    • BP_Shrub.uasset: Responds to “Unarmed” gathering, drops “Food,” and triggers villager gathering animations.

4.3 Dynamic Interaction between AI and the PCG World

The AI system utilizes the following components to understand and manipulate the dynamic world generated by PCG.

  • Environment Query System (EQS)
    • EQC_CollectionTarget.uasset: Defines the query context, e.g., “Find all Actors of type BP_Tree.”
    • EQ_CollectionTarget.uasset: Executes the query to find the nearest and most suitable harvesting target for the AI.
  • Behavior Tree & Blackboard
    • BB_CollectResource.uasset (Blackboard): Acts as the AI’s “memory,” storing information such as the current target (e.g., that specific tree) and carried resources.
    • BT_CollectResource.uasset (Behavior Tree): Defines the AI’s collection logic sequence: Find target via EQS -> Move to target -> Execute harvest animation -> Decrease target health -> Return with resources.

System Linkage: The PCG system generates instances of BP_Tree at runtime. The AI’s EQS system queries these instances in real-time, while the Behavior Tree drives villager characters to interact with them. When a tree is felled (BP_Tree triggers OnResourceDepleted), the Game Mode listens for this event and notifies the PCG system to respawn a new tree elsewhere in the future, forming a complete, dynamic “Generate-Harvest-Regenerate” game loop.

5. Challenges and Solutions

5.1 Performance Optimization Challenges

  • Challenge: Game frame rate dropped significantly when PCG generated thousands of instances.
  • Solution:
    1. LOD Optimization: Set reasonable Levels of Detail (LOD) for all static meshes.
    2. PCG Culling: Utilized distance and frustum culling within PCG graphs to ensure instances are only generated and rendered near the player.
    3. Material Optimization: Merged material textures and used Material Instances to reduce Draw Calls.

5.2 Compatibility between AI Navigation and Dynamic Environments

  • Challenge: Since PCG generates the world at runtime, the AI navigation mesh needs to update correctly to include newly generated buildings and obstacles.
  • Solution:
    1. Used Dynamic NavMesh Bounds Volumes to force a local rebuild of the navigation mesh after building placement.
    2. Fully leveraged the EQS system, allowing AI to dynamically perceive PCG-generated objects during pathfinding rather than relying on predefined waypoints.

5.3 Data Communication between Blueprints and PCG

  • Challenge: How to ensure PCG-generated objects are correctly recognized and interacted with by the Blueprint system.
  • Solution:
    1. Used the Blueprint Spawner node within PCG graphs to directly spawn Blueprint Actors containing game logic.
    2. Used Tags or Interfaces to mark PCG-generated object types, facilitating filtering and interaction for AI and player systems.

7. Minimum System Requirements

  • OS: Windows 10 64-bit
  • Processor: Intel Core i5-7400 or AMD Ryzen 3 1200
  • Memory: 8 GB RAM
  • Graphics: NVIDIA GeForce GTX 1050 Ti or AMD Radeon RX 560
  • Storage: 5 GB available space
